# Audit and Inventory Exports Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add stable audit and inventory export artifacts so external reviewers or developer portals can consume registry state without depending on debug-only traces or live workspace inspection.

**Architecture:** Reuse the generated catalog plus committed provenance artifacts as the source of truth instead of inventing a new server or export database. Add two additive static JSON artifacts, `catalog/inventory-export.json` and `catalog/audit-export.json`, generated by `scripts/build-catalog.sh`, backed by explicit schemas and focused regression tests.

**Tech Stack:** Python 3.11 CLI scripts embedded in `build-catalog.sh`, JSON schemas, script-style regression tests in `scripts/test-*.py`, existing catalog/provenance artifacts, and Markdown docs.

## Scope decisions

- Recommended approach: generate two static export files under `catalog/` so portal, compliance, or reporting tools can consume them from the same committed artifact set already used by the repo.
- Recommended inventory contract: derive it from generated catalog entries plus registry identity so it can expose skills, registries, federation mode, and installability in one stable view.
- Recommended audit contract: derive it from committed provenance plus distribution manifests so it reflects immutable release evidence instead of live release-state checks that depend on the current checkout.
- Rejected approach: create a new `scripts/export-audit.sh` command that recomputes release state from the working tree, because that would make exports depend on mutable local repo state and duplicate the release/provenance pipeline.
- Rejected approach: dump raw `policy_trace` or full dependency planner internals into the export, because those are operator-debug surfaces and not a stable integration contract.
- Rejected approach: fold 11-07 into `catalog/ai-index.json` or `catalog/discovery-index.json`, because those files already serve install/discovery use cases and should not become portal/compliance catch-alls.
- 11-07 should define two export layers:
  - inventory export:
    - registry list with trust/federation visibility
    - skill inventory with source registry identity
    - distribution/installability summary
  - audit export:
    - immutable release/provenance entries
    - delegated review/release context when present
    - exception usage and upstream registry/source snapshot evidence
